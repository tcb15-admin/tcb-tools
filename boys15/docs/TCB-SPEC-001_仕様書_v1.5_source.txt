3. 割振りアルゴリズム
3.1 優先度スコア（低いほど優先）
班内候補の PAST 中央値 median を取り、pastPenalty ＝ sqrt(max(0, PAST − median)) とする。
優先度 ＝ pastPenalty × WA ＋ 累積負荷スコア（当日 score）× WB ＋ 今回割振り数（cnt）× WC
定数 WA=1, WB=1, WC=6（v1.6.4）。中央値以下の PAST にはボーナスを付けない（低累計への集中を防ぐ）。

3.2 PAST（累計負荷）の更新タイミングと単位
割振り実行（btn-run）では PAST を変更しない。confirmCarryOutResult（「この内容で実施確定」）でのみ、表示中の R.map に基づき、担当道具ごとに lscore（重さ・長さから算出、典型 10〜53pt）を PAST に加算する。同一活動日に既存確定があれば旧 map を revert（同 lscore を減算）してから再適用。Cloudflare Worker の confirm-carryout も body.tools を用いて同ロジック。表示は PAST ÷ 15 を四捨五入した「回相当」。旧「回数」単位の PAST が残る場合はシーズン初めの PAST リセットを推奨。

3.3 固定担当の動作
固定担当者も通常割振りの対象に含め、固定道具のスコアを考慮して負荷を分散する。持ち帰り調整時も固定担当は維持。

3.4 両方練習・場所分離
pat が renshu かつ renshuSplit のとき、道具の practiceVenue（マスタ UI の会場列）に応じて A/B へ割当。未設定は従来の試合用 ryo（A/B 列）にフォールバック。

3.5 通常割振り（runAlgo）
活動パターン・欠席・お茶当番・グループに基づき道具を割当。欠席・お茶当番は exSet() により自動割振り候補から除外（手動プルダウンでは選択可）。

3.6 持ち帰り調整モード（runAlgoCarryAdjust）【v1.5／v1.6.5 で「試合前・最小入れ替え調整」へ拡張】
開始経路：STEP1「前回」モーダルの「試合前・最小入れ替え調整を開始」／履歴詳細の同ボタン／STEP3「元に再編集」（beginReeditFromSnap）。UI 上のモード名は v1.6.5 以降「試合前・最小入れ替え調整」。
前回スナップショットの map を基準に、固定担当を維持したまま欠席・お茶当番の変化分をプール再配分。mergePoolSideState でお茶当番引き継ぎ後の side 状態を fullSt に反映。
活動パターンが前回と不一致でも map ベースで調整可能（バナーで案内）。活動日は開始時に本日へ設定。
前当番が基準メンバー集合にいない道具は引き継ぎ0件になりうる（STEP3 手動可）。
再編集開始時に LS_PREV を上書きしない（v1.6.3：persistAllocationSnapshot 削除）。

3.6.1 現地の実保有に補正（openActualHoldModal / saveActualHoldAdjust）【v1.6.8】
背景：前回割振り確定後、現地での受け渡しやマスタの道具固定機能により、「前回確定（PAST／seedBaseSnap）」と「保護者が実際に持っている道具」がズレることがある。この状態のまま調整を始めると、LINE／PDF の入れ替え案内が実態と食い違う。
動作：試合前・最小入れ替え調整のバナー（p1/p2-seed-mode-banner）の「現地の実保有に補正」ボタンで補正モーダル（actual-hold-modal）を開く。前回の割り当て（seedBaseSnap.map）を道具ごとに一覧し、各道具の担当を「いま実際に持っている人」へプルダウンで直せる（変更行は橙 tcb-actual-hold-changed）。
保存（saveActualHoldAdjust）：補正内容を seedBaseSnap.map に反映し、carryAdjustRefSnap（差分レポートの基準）も同じ実態へ更新する。以後の「前回持っていた道具」の基準と、3.7 の差分レポート（＝LINE／PDF の入れ替え案内）はこの実態基準で生成される。補正は開始時点の基準のみを変え、割振りアルゴリズム本体（3.1〜3.7）には手を入れない。

3.7 試合前・最小入れ替え調整の最小変更ロジック【v1.6.5〜v1.6.7】
狙い：朝の集合が班別で道具の物理的入れ替えができない前提のもと、前回の実施確定を基準に「同じ人・同じ道具」を最大限維持し、保護者の入れ替え負担（＝担当が変わる人数）を最小化する。
(1) ベース維持（buildPrevSeedBaseAndPool）：班・道具種別が前回担当と整合する道具は前回担当（baseMap）を維持。整合しない道具のみプール（poolSet）へ。
(2) お茶当番引き継ぎ（applyOchHandoff）：前当番の担当道具を現当番へ移す。受け手不在時はプールへ。
(3) 班入替の2人スワップ（applyMinimalCrossTeamSwaps／team モードのみ）：プール道具のうち、前回担当が現在の相手班にいるものを2人1組で相互交換し、変更人数を増やさずに班整合を回復。成立ペアは baseMap へ戻し poolSet から除外。
(4) 残りプールの集中配分（assignPoolCarryAdjust → assignOneToolToState の minChangers オプション）：余った道具は「担当を広く分散」ではなく「すでに変更が必要な人」へ集中させる。優先順は概ね ①前回欠席→今回参加（zeroPriority）②既にプール道具を受けた人（poolAssigned）③既に前回と持ち道具が変わった人 ④前回その道具の担当者。前回から変化のない人は大幅に後回し（実質的に温存）。1人が複数道具を持つことを許容し、入れ替え対象の家族数を減らす。
(5) 制約：車両積載上限（capHv/capSL/capLlen）・単独持ち帰り（noMul／toolSoloExclusiveByRow）は従来どおり順守。固定担当は変更しない。
差分レポート（buildCarryAdjustChangeReport／formatCarryAdjustChangeText）：基準 map と現 map を比較し「維持点数／変更点数」「2名で道具を交換（相互スワップ）」「その他の担当変更（片方向）」「持ち帰り内容が変わる方＝前回・今回の持ち道具一覧」を整理。STEP3 の紫サマリーに表示し、コピー・PDF 2ページ目・LINE 用メッセージへ出力。

4. 主要機能一覧
4.1 STEP1（活動設定）
活動パターン、活動日、グループ名、メンバー未発表、前回モーダル、試合前・最小入れ替え調整の開始

4.2 STEP2（欠席・当番）
対象外選択、お茶当番、道具一時除外、グループ振り分け（試合モード）、割振り実行。試合前・最小入れ替え調整時は STEP1/2 に青バナー表示（「現地の実保有に補正」ボタンから実保有補正モーダルを起動可）

4.3 STEP3（割振り結果）
道具別表（複数担当強調・会場分離時の列）、メンバー別タブ、印刷/PDF タブ（LINE 用メッセージコピー含む）、実施確定カード、Undo/Redo、元に再編集、班跨ぎ手動（chk-manual-cross-team・手動プルダウンのみ）、公平に再分配（UI_SIMPLE 時非表示）。試合前・最小入れ替え調整時は入れ替え案内サマリー（p3-carry-adjust-summary：維持行=緑 p3-carry-keep-row／変更行=橙 p3-carry-changed-row の色分け、テキストコピーボタン）を表示

4.3.1 印刷/PDF（buildPrintHTML）
人数に応じて 3〜4 列で組み、道具名を強調（色・nowrap）・道具間余白を確保し A4 1ページに収める密度制御。試合前・最小入れ替え調整で入れ替えがある場合のみ、入れ替え案内（formatCarryAdjustChangeText）を PDF 2ページ目として追加（@page/height:auto・overflow:visible に切替）。LINE 用メッセージ（copyAssignShareMessage）にも入れ替え案内テキストを付加

4.4 割振り履歴
一覧・詳細モーダル、試合前・最小入れ替え調整の開始、直近1年 PDF（UI_SIMPLE 時非表示）、JSON 書出、選択削除、全削除

4.5 マスタ設定
メンバー・道具の DnD、DESCS 連動、パスワード変更、会場列、Cloudflare 保存。メンバー対象外は 2×2 グリッド（休部 rest／コーチ coach／14期帯同 ac14／兄弟所属 sibling）で指定し、いずれか ON のメンバーは割振り対象・STEP1/2 の候補から常時除外（バッジ ac14-badge・sibling-badge で識別）。道具説明は全幅 textarea で編集

7. セキュリティ（運用上の注意）
SYNC_API_TOKEN は Worker secret と HTML 内 Bearer で一致させる。Git への平文コミットは避け build 時注入を推奨。クライアント埋め込み Bearer は URL 入手者がソースから読み取りうる（Cloudflare Access 等は将来検討）。
