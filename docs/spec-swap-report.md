## 1. 目的
保護者間で合意した道具担当の交代を、保護者がアプリから**結果報告**し、道具MGRが内容を確認して**割振りへ反映（または却下）**できるようにする。連絡の取りこぼしと手作業を減らす。

## 2. 用語
- **保護者確認画面**: 世代共通の公開URL（無認証・読取ベース）で開く画面（`kakunin.html?v=<shareId>`）。
- **交代報告**: 保護者が申請する「道具X: 現担当A → 新担当B」の結果報告。
- **道具MGR / ツール**: `tool_template.html`（管理者が操作するアプリ）。
- **反映**: 報告内容を当日の割振りデータに適用するMGR操作。
- **修正版再通知**: 反映後の最新割振りを、同一URLで再公開し、LINEで再案内すること。

## 3. スコープ
- 保護者側: 氏名選択・自分/全員トグル・交代報告フォーム・受付状況表示。
- MGR側: 新着バッジ📮・Web Push受信・一覧/詳細・反映/却下（プリセット理由）・検証警告・修正版LINE再通知。
- バックエンド: 報告の保存/取得/状態更新、push購読の保存/送信、公開日の再公開（同一URL維持）。

## 4. 決定事項サマリ

### 4-1. 確定済み（前提）
- Web Push: **MGRのみ・最初から実装**（Web Push＋VAPIDで**追加費用ゼロ**）。

## 5. 保護者確認画面の仕様

### 5-1. 氏名選択と表示フィルタ
- 初回アクセス時に**氏名ピッカー**（今回公開データの氏名一覧）を表示。選択でその人の担当**のみ**表示。
- 選択は**端末に記憶（localStorage）**。次回以降は自動で絞り込み。「別の人に切替」も用意。
- 記憶した氏名が今回公開に存在しなければピッカーへ戻す（世代/公開更新で氏名変動に対応）。

### 5-2. 表示トグルとハイライト
- 上部に「**自分のみ / 全員を表示**」トグル。
- 全員表示時、自分のカードを **navy枠＋「あなた」バッジ**で強調。
- 新規CSSは `pv-` プレフィクスで既存セレクタと非衝突。

### 5-3. 交代報告フォーム（各道具行にインライン）
- 表示: `道具X ／ 現担当A（自動・読取専用）／ 新担当 [▼ 既定「—」]`
- 現担当Aは**公開データから自動確定**（改ざん不可）。**誰の視点でも起票可**（①）。
- **新担当ドロップダウンの候補（⑩）**: **当日の割振り対象メンバー**を表示。
  - #1 欠席・#2 恒久対象外（休部・退部・コーチ・14期帯同・兄弟所属）は**候補に出さない**（構造的に排除）。
  - #4「担当0件」の余りメンバー（参加・有効だが今回は道具が回らなかった人）は**選択可**。
  - #3 お茶当番は**「当番」ラベル付き**で候補に含め、可否の最終判断はMGRの反映時検証に委ねる。
  - 自分（現担当A）は候補から除外。
- 新担当に **A以外**を選ぶと「**申請**」ボタンが活性化。
- **申請前の事前防止（オペミス・無効申請を入口で弾く）**:
  - **確認ダイアログ**「道具Xを A→B で交代申請します。よろしいですか？」→ OK/キャンセル（選択ミス抑止）。
  - **現担当Aの鮮度チェック**: 送信時にサーバで最新割振りと照合し、既に別人へ変わっていれば「この道具の担当は既に○○です。最新をご確認ください」と弾く（古い内容の申請防止）。
  - **班（A/B）またぎ注意**: AとBが別班なら「別の班です。会場が分かれる可能性があります。よろしいですか？」と注意表示（ブロックはしない）。
  - **重複申請の抑止**: 同一道具に同内容の未処理報告があれば「既に申請済みです」と抑止。
- **「申請せずに閉じる」**を明示（変更しても送らない選択肢）。
- **コメント欄**（申請ボタン付近・任意）: **上限100字・文字数カウンタ付き**。
- **連絡者**: **氏名一覧から選択**（任意、既定は選択中の「自分」）。

### 5-4. 受付状況表示（却下・進捗のフィードバック＝プル型）
- 自分に紐づく過去の報告の状態を表示: **未対応 / 反映済み / 見送り（＋却下理由）**。
- 却下時は**却下理由（プリセット文＋自由記入）**まで表示。
- 保護者は画面を再度開くと状況と理由を確認できる（プッシュはしない）。

## 6. 交代報告フロー（保護者）
1. URLを開く → 氏名選択（or 記憶）。
2. 対象道具の新担当を選択 → コメント/連絡者（任意）→「申請」。
3. 確認ダイアログ → OK → サーバ保存 → 「受け付けました」表示。
4. 以後、受付状況で結果（反映/見送り＋理由）を確認。

## 7. 道具MGR側の仕様（ツール）

### 7-1. 新着通知
- ヘッダーに **📮 交代報告（N）** バッジ（プル型・常設）。
- 加えて **MGRのみ Web Push**: 新着報告時に端末へ通知（許可制）。未許可/非対応時はバッジにフォールバック。

### 7-2. 一覧・詳細
- 報告一覧: 日付・道具・A→B・連絡者・コメント・状態。
- **同一道具に複数報告がある場合は新しい順に並べ、競合を強調表示**（⑤）。
- 詳細で「**反映**」「**却下（理由選択）**」を選択。

### 7-3. ワンタップ反映（定義）
> 保護者はアプリ**外**で交代を合意済み。アプリは**結果報告**を受け、MGRが承認して反映するだけ。

- 報告詳細の「**反映**」1タップで:
  1. **検証**（§8）→ 問題あれば警告表示＋続行確認。
  2. 当日の割振りデータで **道具Xの担当を A→B に置換**。メモリ上に STEP3 結果（`R`）が無い場合は、報告日の**履歴（実施確定優先）または LS_PREV**から割振りを復元して STEP3 を開いてから置換する（「割振りを実行」のやり直しは不要）。
  3. 報告を**反映済み**に更新、履歴へ記録。
  4. STEP3 を表示し、「実施確定」→展開情報から修正版LINE再通知するよう案内する。
- **マイルストーン操作（反映／却下／一括反映）は必ず確認ダイアログ**を挟む。

### 7-4. 却下（プリセット理由）
- 「却下」時に**理由を選択（必須）**＋任意で自由記入（②）。
- 方針: 機械的に判定できるケース（班またぎ／欠席・対象外／古い内容）は**§5-3の事前防止と§8の反映時警告で潰す**ため、却下理由は**人の判断のみ**に絞る。
- プリセット却下理由（案・調整可）:
  - D1: 交代は取りやめになった（申請の取り下げ）
  - D2: 内容を確認できない／要再連絡
  - D9: その他（自由入力）
- 報告を**見送り**に更新し、理由コード＋文言を保存 → 保護者の受付状況に表示（§5-4）。
- 補足: 稀な「交代不可の道具（重量物・特定担当）」は道具マスタに専用フラグを持たないため機械判定不可。**現状は未対応**とし、来た場合は D2/D9 で対応。将来必要なら道具マスタに「交代不可」フラグを追加し、その道具は新担当欄を無効化（申請自体をさせない）で対応可能。

### 7-5. 修正版のLINE再通知（新規）
- 設置場所: **反映直後の導線＋「保護者へ共有」ブロックに常設ボタン**（⑧）。
- ワンタップで:
  1. **再公開**（現在の確定データを**同一URL**で更新）。
  2. **「修正版」用LINE本文を自動生成してコピー**（＋任意で共有シートを開く）。
- 本文テンプレート（⑦・5行前後・要点のみ／**今回反映した交代のみ**を列挙＝⑨）:

  ```
  【修正版】M/D(曜) 道具割り振りを更新しました
  交代を反映しました：
  ・「バット」中村 → 田中
  ・「ヘルメット」佐藤 → 鈴木
  最新はこちら（同じURLで最新版に更新されています）
  https://…/kakunin.html?v=xxxx
  ※各自の詳しい担当は確認画面またはPDFをご覧ください。
  ```
  - 反映が無い更新なら「内容を更新しました」に切替。
  - 日付・URLを自動差し込み。既存の共有本文（`SHARE_PDF_MESSAGE`）とトーンを統一。
- 既存関数を再利用（`publishParentView` / `copyTextToClipboard` / `shareAssignPdf` / `formatCarryAdjustChangeSummary`）。実装は軽量。

## 8. 検証・警告ルール

### 8-1. 事前防止（申請フォーム側＝入口で弾く／MGR手数の削減が目的）
- **新担当候補を当日の割振り対象メンバーに限定** → #1欠席・#2恒久対象外は選べない（⑩・§5-3）。
- **現担当Aの鮮度チェック**（サーバ照合）→ 既に変わっていれば送信を弾く（古い内容の申請防止）。
- **確認ダイアログ**（A→B）→ 選択ミス（オペミス）抑止。
- **班（A/B）またぎ注意表示**（ブロックはしない）。
- **重複申請の抑止**（同一道具・同内容の未処理報告）。

### 8-2. 反映時の警告（MGR側＝最終判断の補助）
- **現担当がAでない**（申請後に変わった）→ 警告＋上書き確認。
- **Bが反映時点で 欠席／対象外に変化** → 警告（それでも反映するか確認）。
- **A/B班をまたぐ交代** → 注意表示。
- **同一道具に複数報告** → 新しい順＋競合提示（⑤）。

補足: お茶当番（#3）は候補に「当番」ラベル付きで出るため、反映時にMGRが可否を最終判断する。

## 9. Web Push 仕様（MGRのみ・無償）
- **Service Worker** を追加しGitHub Pagesに配置（スコープはリポジトリ配下）。
- **VAPID鍵ペア**: 秘密鍵は Worker Secret、公開鍵はクライアント埋め込み。
- MGRが**通知許可** → 購読情報を D1 に保存。
- 新着報告時、**Worker が Web Push 送信**（VAPID JWTをWebCryptoでES256署名）。
- iOS制約: iOS 16.4+ かつ**ホーム追加したPWA**でのみ受信・許可必須・配信は最善努力（遅延あり得る）。
- 追加費用なし（ブラウザ各社のpushサービス・VAPIDは無料、Apple Developer Program不要）。

## 10. データモデル（案・D1）

```sql
CREATE TABLE IF NOT EXISTS swap_reports (
  id TEXT PRIMARY KEY,
  gen TEXT NOT NULL,            -- 世代 (boys15 等)
  share_id TEXT NOT NULL,       -- 公開日ID
  day_key TEXT NOT NULL,        -- 対象日
  tool TEXT NOT NULL,           -- 道具名
  from_person TEXT NOT NULL,    -- 現担当A（サーバで公開データと照合・自動確定）
  to_person TEXT NOT NULL,      -- 新担当B
  reporter TEXT,                -- 連絡者（メンバー氏名・選択式・任意）
  comment TEXT,                 -- 100字以内
  status TEXT NOT NULL DEFAULT 'pending', -- pending/applied/dismissed
  reject_code TEXT,             -- 却下理由コード（D1/D2/D9）
  reject_reason TEXT,           -- 却下理由文（自由記入）
  created_at INTEGER NOT NULL,
  handled_at INTEGER
);

CREATE TABLE IF NOT EXISTS push_subscriptions (
  id TEXT PRIMARY KEY,
  gen TEXT NOT NULL,
  endpoint TEXT NOT NULL UNIQUE,
  p256dh TEXT NOT NULL,
  auth TEXT NOT NULL,
  created_at INTEGER NOT NULL
);
```

## 11. API エンドポイント（案）
- `GET /api/public/day`（既存・無認証）: 公開データ取得。**当日の割振り対象メンバー一覧（氏名＋お茶当番フラグ）を最小限追加**し、新担当ドロップダウンの候補に使う（⑩）。
- `POST /api/public/swap-report`（無認証・保護者用）: 報告作成。サーバ側で share_id 有効性・現担当A鮮度照合・新担当B∈対象メンバー・重複抑止・入力検証・レート制限。
- `GET /api/public/swap-status`（無認証）: 自分（氏名）に紐づく報告状態の取得。
- `GET /api/swap-reports`（要トークン・MGR）: 一覧。
- `POST /api/swap-reports/handle`（要トークン・MGR）: 反映/却下＋理由コード/文言。
- `POST /api/push/subscribe`（要トークン・MGR）: 購読登録。

## 12. セキュリティ（無認証投稿対策）
- 入力検証: 道具/現担当A/新担当B/連絡者は**公開データ・当日の割振り対象メンバーに存在する値のみ許可**、B≠A、コメント100字上限、型・長さ制限。
- **現担当Aの鮮度照合**（サーバ側で最新割振りと突合し、古い内容は拒否）。
- **share_id の有効性チェック**（未公開/失効IDは拒否）。
- **レート制限**（IP/share_id 単位で連投抑制）＋簡易スパム対策。
- 保存時 **XSSエスケープ**、表示時サニタイズ。数値/フラグは `(int)` キャスト方針を踏襲。
- push秘密鍵は Secret 管理。公開エンドポイントは最小権限。
- 公開データへの対象メンバー一覧追加は**氏名＋当番フラグのみ**の最小限に留める（既に割振り結果で大半の氏名は公開済みのため増分リスクは軽微）。

## 13. 段階リリース
- **Phase 1**（実装済）: 保護者フォーム＋MGR一覧/反映/却下（プリセット理由）＋受付状況＋修正版LINE再通知＋プル型バッジ。
- **Phase 2**（実装済）: Web Push（MGRのみ）。`sw.js`＋`tcb-push-mgr.js`＋`/api/push/subscribe`＋Worker 送信。
- リリース手順: D1マイグレーション → Worker deploy → 静的サイト push（`cloudflare-sync/README.md` §13–14）。

## 14. 非対象（やらないこと）
- 保護者間の合意調整そのもの（アプリ外）。
- 保護者への push 通知（多数・無認証のため対象外。却下等はプル型で提示）。

## 15. 既知の留意点
- 反映後に保護者画面へ反映するには**再公開が必要**（同一URL維持のため配布済みURLはそのまま最新化）。
- 過去に確認済みの `confirmedAt` は、STEP3を再実行して再確定しないと欠落し得る既存挙動。反映→再確定→再公開の順を推奨。

## 16. 実装計画

### 実装方針
- **JS/CSSは独立ファイル**に分離し、役割別に部品化する（既存 `tcb-print-pdf.js` / `tcb-sync-api.js` と同様に `build.py` で配布に取り込む）。
- **CSSセレクタは新プレフィクス**（保護者側 `pvsw-`／MGR側 `tcbsw-`）で既存と非衝突。
- **DBはSQLを用意するのみ**（実行はユーザー承認後）。冪等な `CREATE TABLE IF NOT EXISTS`。
- **基本セキュリティ**（入力検証・A鮮度照合・レート制限・XSSエスケープ・最小権限）を各層で担保。
