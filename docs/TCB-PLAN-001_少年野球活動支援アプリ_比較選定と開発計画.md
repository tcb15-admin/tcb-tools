## 0. 確定方針（2026-07-15）

| 項目 | 決定 |
|------|------|
| 対象 | 東海中央ボーイズ **15期の拡張**（一般向け SaaS 化はしない） |
| TeamHub／PLAY | **導入しない**。機能・優先度の**参考比較のみ** |
| 自作の最優先 | **①出欠・連絡** → **②配車（車出し）** |
| アプリ構成 | **機能別アプリ＋役割ポータル**（§3A／§3B） |
| 技術 | **既存延長を基本**（静的 HTML＋Cloudflare Worker／D1＋GitHub Pages＋PWA）。Next.js は条件付きで後検討（§4） |
| スコア本丸 | **将来やりたい**。難易度が高いため優先度は後ろ。出欠・配車の安定運用後 |
| 連絡チャネル | 保護者との日常やりとりは **LINE**。アプリは「回答画面・一覧・催促テキスト生成」に徹する |

### 結論（いま進めること）

1. TeamHub は入れないので、**出欠回収は自作が必須**になる。  
2. 既存の **保護者公開ページ（`publish-day`／`/api/public/`）＋ LINE で URL を配る**パターンを、出欠・配車に再利用する。  
3. **1つの巨大アプリに詰めず**、出欠／配車／道具を分けて作り、**活動（activity）ID と共有マスタ**でつなぐ。  
4. 入口は **ポータル**。各役割マネージャーは自分のアプリへ直行（ブックマーク可）。  
5. 出欠 MVP → 配車 MVP の順で正式運用。作戦盤・QR・スコアは後段。

## 3A. 機能別アプリ分割とデータ連携（推奨）

### 結論

**可能です。むしろ 15期向けには「機能別に分けて、共通マスタ＋活動データで連携」する構成を推奨します。**  
いわゆるマイクロサービスまでは不要で、**フロントを機能ごと分割／バックエンド（Worker＋D1）は共通**が現実的です。

### なぜ分けるか

| 利点 | 内容 |
|------|------|
| 段階リリース | 出欠だけ先に本番。配車は後から足せる |
| 画面肥大の回避 | 道具割り振りの巨大 HTML に全部を突っ込まない |
| 役割の明確化 | 道具 MGR／出欠担当／配車担当で使う入口を分けられる |
| 障害の局所化 | 配車の不具合が出欠回収を止めにくい |

### 全体像

```text
                    ┌─────────────────────────────┐
                    │  Cloudflare Worker + D1      │
                    │  （唯一のデータの正）           │
                    │  ・master（メンバー・道具）     │
                    │  ・activities（活動）          │
                    │  ・attendance（出欠）          │
                    │  ・carpool（配車）             │
                    │  ・carryout / history（道具）  │
                    └──────────────┬──────────────┘
                                   │ 同一 API・同一 cohort=15
         ┌─────────────┬───────────┼───────────┬─────────────┐
         ▼             ▼           ▼           ▼             ▼
   [マスタ編集]    [出欠アプリ]  [配車アプリ]  [道具割振り]   [保護者公開]
    共通 or 既存     新規         新規         既存継続      URL 単位
```

配信はいずれも GitHub Pages 上の静的ページでよい例:

| アプリ | URL 例 | 主データ |
|--------|--------|----------|
| ハブ（入口） | `/boys15/` または `/boys15/hub/` | 各アプリへのリンク |
| 道具割り振り | `/boys15/`（現行） | master＋割振り履歴 |
| 出欠 | `/boys15/attendance/` | activities＋attendance |
| 配車 | `/boys15/carpool/` | carpool（出欠参照） |
| 保護者回答 | `/boys15/p/` 等 | shareId 公開 API |

共通化する部品: `tcb-sync-api.js`、認証（パスワード／Bearer）、メンバー表示、LINE テキスト生成ヘルパー、テーマ CSS。

### マスタ共有

| データ | 正の置き場 | 参照するアプリ |
|--------|------------|----------------|
| メンバー（氏名・班・対象外フラグ等） | 現行 `tool_state.master_json`（または将来 `members` 表） | 出欠・配車・道具・作戦盤 |
| 道具マスタ | 同上 | 道具・配車の積み込み |
| パスワード／世代設定 | config＋API | 全スタッフィアプリ |

**原則**: マスタは **1か所だけ編集**（現行のマスタ画面、または薄い共通マスタアプリ）。他アプリは `GET /api/state` 相当で読む。  
出欠アプリ内でメンバーを増減しない（二重管理を防ぐ）。

### 活動（activity）を横断キーにする

出欠・配車・道具は「日付の文字列」だけだと曖昧になりやすい（午前練習＋午後試合など）。  
そのため **活動 ID（activity_id）** を共通のひも付けにする。

```text
activity_id = 2026-07-20-practice-am  など
  ├─ attendance_responses（誰が出／欠）
  ├─ carpool_*（誰が車出し・同乗）
  └─ tool allocation / confirm-carryout（誰が何を持つ）
```

道具割り振りの現行「活動日」は、移行期は `activity_date` のまま動かし、**出欠連携時に activity_id を渡せる**ように拡張する（破壊的変更を一度にやらない）。

### 出欠 → 配車 → 道具の連携ルール

| 連携 | 動き |
|------|------|
| 出欠 → 配車 | 配車画面は `status=in`（出席）のみ候補。欠席・未回答は乗車／運転から除外（未回答は警告表示） |
| 出欠 → 道具 | STEP2 の欠席リストを、出欠 API から **ワンタップ取込**（手動チェックと併用可） |
| 配車 → 道具 | 積み込み備品を割振り結果と突合（「この車にテント」など）。Phase 3 |
| マスタ → すべて | 休部・退部・対象外は全アプリで候補除外 |

**書き込みのオーナー**（競合を避ける）:

| データ | 書くアプリ | 他アプリ |
|--------|------------|----------|
| master | マスタ／道具側 | 読み取りのみ |
| attendance | 出欠（＋保護者公開 API） | 配車・道具は読取 |
| carpool | 配車のみ | 他は読取 |
| 割振り map | 道具のみ | 他は読取 |

### 実装形態の選択肢

| 案 | 内容 | 向き |
|----|------|------|
| **A. マルチページ（推奨）** | 機能ごとに HTML＋JS。API 共通 | 15期・既存延長と最も相性が良い |
| B. 単一 SPA 内のルート分割 | 1 つのフロントで `/attendance` 等 | 将来 Next.js 移行時 |
| C. 完全別リポジトリ | メンテ・トークン・デプロイが分散 | **非推奨**（チーム規模に不釣合い） |

**Next.js との関係**: 機能分割そのものは静的マルチページで足りる。後から出欠だけ Next 化する、といった部分移行も、**API が共通なら可能**。

### 注意点

1. **同期タイミング** — 出欠を直した直後に道具側が古い欠席のまま、を防ぐため「取込ボタン」か「開始時に API 再取得」を明示する。  
2. **権限** — 当面は世代パスワード共有のままで可。将来、機能ごとに権限を分けるなら API トークン／ロールを検討。  
3. **オフライン** — 道具は localStorage キャッシュあり。出欠の正はサーバ側。衝突時は「サーバ優先＋上書き確認」。

## 3B. 役割ポータルとマネージャー運用 UI（推奨）

### 結論

提示のイメージ——**ポータルがあり、各役割のマネージャーが自分のアプリだけを使う**——は **良い設計**です。15期の分業（出欠／道具／配車／お茶当番など）と一致します。  
そのうえで、実務で効く追加アイデアを次に固定します。

### 推奨 UI 構造

```text
[15期ポータル]  ← 全員の入口・「直近の活動」ダッシュボード
    │
    ├─ 出欠マネージャー用 → 出欠アプリ
    ├─ 配車マネージャー用 → 配車アプリ
    ├─ 道具マネージャー用 → 道具割り振り（現行）
    ├─ お茶当番マネージャー用 → お茶当番アプリ（薄く）
    ├─ （将来）スコア／作戦盤 …
    └─ マスタ編集 → 原則 1 か所（メンバー・道具）※権限は別途
```

保護者はポータルに入らない。 **LINE で届いた公開 URL だけ**を使う。

### ポータルに置くもの（カードではなく「今日やること」中心）

ダッシュボード化（統計だらけ）は避け、**直近の活動 1〜2 件**と役割への入口に絞る。

| 要素 | 内容 |
|------|------|
| 直近の活動 | 日時・場所・種別。全役割で同じ「いま対象の活動」が見える |
| 役割タイル | 出欠／配車／道具／お茶… 大きめタップ領域。自分の担当へ直行 |
| 進捗ランプ | 例: 出欠「回答 18/22」・配車「未作成」・道具「未確定」・お茶「佐藤家」 |
| 次アクション | 例: 「出欠が揃ったら配車へ」など、依存関係を一文で |
| 各アプリ直リンク | ホーム画面に追加／ブックマーク用。毎回ポータル必須にしない |

### 役割とアプリの対応（案）

| 役割 | アプリ | 主な仕事 | 他役割との接点 |
|------|--------|----------|----------------|
| 出欠 MGR | 出欠 | 活動の出欠回収・催促・締め | 出席確定を配車・道具・お茶が参照 |
| 配車 MGR | 配車 | 車出し・乗車・積み込み | 出席者のみ。積み込みは道具と連携 |
| 道具 MGR | 道具（既存） | 割振り・実施確定・交代承認 | 欠席・お茶当番を取込 |
| お茶当番 MGR | お茶（薄型） | 当番家庭の決定・交代・LINE 案内 | 道具 STEP2 のお茶当番へ反映 |
| （任意）全体／監督 | ポータルのみでも可 | 進捗確認・差し戻し依頼 | 各アプリの「閲覧」 |

**お茶当番は独立フルアプリにしない方がよい場合がある。**  
「当番を決めて LINE し、道具側に渡す」だけなら、**出欠または活動詳細の中の小さなパネル**でも足りる。役割担当者が別なら、**薄い専用画面**にしてポータルから出す、がバランス良い。

### 権限の段階（やりすぎ防止）

少年野球の運用では、最初からユーザーアカウント＋RBAC は重い。次の段階を推奨する。

| 段階 | 内容 | いつ |
|------|------|------|
| **0（開始時）** | 世代の共通 PW。ポータルで「自分の役割」を選ぶと次回からそのアプリを優先表示（端末に記憶） | Phase 1 |
| **1** | 役割ごとに短い **役割 PIN**（出欠用／道具用…）。間違ったアプリを触れなくする（完全保密ではない） | 分業が定着したら |
| **2** | 個人アカウント／監査ログ | 運用者が増える・トラブルが増えたら |

ポータルの「役割を選ぶ」は **UX の近道**であり、セキュリティの本丸ではない（その前提をマニュアルに書く）。

### 活動パイプライン（役割をつなぐ合図）

全マネージャーが同じ活動を見て、依存順が分かるようにする。

```text
① 活動作成（出欠 or ポータル）
 → ② 出欠回収・締め（出欠 MGR）
 → ③ お茶当番確定（お茶 MGR）
 → ④ 配車作成（配車 MGR）
 → ⑤ 道具割振り・実施確定（道具 MGR）
 → ⑥ LINE 展開（各 MGR が自分の担当文／または道具の統合展開）
```

**ハードロックはしない**（現場は順序が前後する）。代わりに:

- 配車画面: 「出欠未締め」なら黄警告（操作は可能）  
- 道具画面: 「出欠から取込」ボタン＋取込時刻表示  
- ポータル: ⑤の前に ②が赤なら「先に出欠を」と表示  

### 追加で効くアイデア

1. **ホーム画面ショートカットを役割ごと**  
   出欠 MGR は出欠 PWA、道具 MGR は現行道具 PWA。ポータルは新任・監督用。
2. **「依頼を出す」ボタン（軽量）**  
   例: 道具 MGR がポータルから「出欠の催促お願いします」→ 出欠用の催促テキストが用意される／Web Push（既存基盤流用）。チャットルームは作らない。
3. **マスタ編集は限定**  
   メンバー増減は道具 or 専用マスタ。出欠・配車 MGR は基本触らない（誤編集防止）。
4. **交代・代理**  
   ポータルに「今シーズンの役割担当者」を表示（誰に聞けばよいかが一目）。データ上の必須ではないが運用で効く。
5. **保護者向けは役割を見せない**  
   「出欠回答」「交代報告」など用途 URL のみ。

### 避けたい形

| 避けたい | 理由 |
|----------|------|
| 全機能入りの巨大管理画面 | 担当外の操作ミス・画面が重い |
| 役割ごとに別パスワードを最初から複雑化 | 引き継ぎコストが高い |
| ポータル必須・アプリ直 URL 禁止 | 毎回の手間が増え、使われなくなる |
| 役割間チャットの自作 | LINE がある。依頼テンプレで足りる |

### ポータルの初期スコープ（実装順）

1. Phase 1 と同時または直後: **リンク一覧＋直近活動名**だけのポータル  
2. 出欠が動いたら: **回答進捗ランプ**  
3. 配車・お茶追加時: タイルとパイプライン表示を増やす  
4. 役割 PIN は需要が出てから

## 3. 15期向けロードマップ（確定優先度）


### Phase 1 — 出欠・連絡 MVP（最初に正式運用）

**目的**：毎週の「誰が来る／来ない」回収と、未回答への催促コストを下げる。

| 役割 | できること |
|------|------------|
| スタッフ（MGR） | 活動（日時・場所・種別）を登録。LINE 用の案内文＋回答 URL を生成。回答一覧・未回答一覧。催促用テキスト |
| 保護者 | スマホで URL を開き、選手単位で 出／欠／未定 を回答（コメント可）。後から変更可 |

**既存資産の再利用**

- `POST /api/publish-day` + `GET /api/public/day` と同型の **shareId 公開ページ**
- LINE 展開パネル（現行の Web Share／テキストコピー）
- メンバーマスタ（`master` の選手一覧）を出欠対象に流用
- （任意）未回答リマインドはまず **催促テキストのワンタップ生成**。自動プッシュは Phase 1.5

**Phase 1 でやらないこと**

- アプリ内チャット、既読管理の完全再現
- 成績・スタメン・配車の本実装
- 保護者アカウント登録（メール／パスワード）— **URL＋選手選択（または簡易 PIN）**で足りるところまで

**成功条件の例**

- 試合／練習のたびに、スタッフが出欠を開き LINE で送れる  
- 保護者が 30 秒以内に回答できる  
- 未回答者リストから催促文をコピーできる  

### Phase 2 — 配車（車出し）MVP

**目的**：車出し担当・乗車人数・積み込み備品の可視化と、役割の偏りの見える化。

| 項目 | 内容 |
|------|------|
| 入力 | 車出し可否、乗車可能人数、実際の乗車メンバー、積み込む大物（テント等・道具マスタ連携可） |
| 表示 | 車両単位の一覧（誰が運転・誰が同乗・何を積む） |
| LINE | 配車表テキストのワンタップ生成 |

**出欠との接続**：欠席者は配車候補から外す。出欠確定後に配車を始める導線。

**Phase 2 でやらないこと**：最適配車アルゴリズム、地図 API、有料ライドシェア連携。

### Phase 3 — 既存道具割り振りとの連携強化

- 配車の「積み込み」と割振り結果の突合  
- 必要なら所持ステータス／QR（後回し可）

### Phase 4 — 作戦盤・LINE 整形の拡張

- DnD 守備配置、スタメンの LINE 用整形（出欠で「出席」の選手だけ候補）

### Phase 5 — スコア本丸（将来・難易度見合い）

- Play-by-Play 級は工数が大きい（入力 UI・ルールエンジン・訂正・スタッツ）  
- 最初の現実的ステップは **ボックススコア簡易** → スタッツ集計  
- 球数・休養アラートはスコアより先に「手動記録＋設定 JSON」でも価値が出る

## 4. 技術方針：既存延長 vs Next.js（プロ意見）

### 結論

**出欠・配車（Phase 1〜2）は既存スタック延長を推奨。いま Next.js に乗り換える必要はない。**  
スコア本丸や画面数が急増した段階で、**モジュール単位の導入または部分移行**を検討する。

### 理由（既存延長が勝つ点）

1. **すでに「保護者がトークンなしで触る公開 API」がある**  
   出欠・配車の核心パターン（スタッフが発行 → LINE で URL → 保護者が回答 → D1 に保存）が `published_days`／`swap_reports` で実証済み。
2. **15期専用・メンテ担当が続く前提**では、ビルドパイプラインを増やさない方が運用が軽い（現行は `build.py`＋Worker）。
3. 出欠・配車は **CRUD＋一覧＋LINE テキスト**が主で、React/SSR の恩恵が小さい。
4. PWA・Web Push・GitHub Pages 配布が既にある。

### Next.js を検討すべきタイミング（トリガー）

次の **どれか複数**が当てはまったら検討する。

| トリガー | なぜ |
|----------|------|
| スコア本丸（リアルタイム入力・複雑な状態） | コンポーネント分割・状態管理の必要性が跳ねる |
| 画面・機能が「道具＋出欠＋配車＋スコア」で単一 HTML が肥大化 | 開発・衝突・テストコストが増える |
| TypeScript 必須・複数人開発 | 型とモジュール境界のメリット |
| SEO やサーバ描画が必要 | 15期内部ツールでは通常不要 |

**採用する場合の現実的な形**

- いきなり全体を Next.js 化しない  
- 例: `attendance/` だけ Vercel＋Next、API は **同じ Cloudflare Worker** を呼ぶ  
- または Worker 側は維持し、フロントだけ段階移行  

**今はやらなくてよいこと**

- Supabase 新設（D1 が既にある）  
- 出欠のためだけに Next.js＋DB を二重化  

### 推奨スタック（Phase 1〜2）

| 層 | 採用 |
|----|------|
| UI | `template/` 延長（出欠・配車用の画面／JS を部品分割） |
| API | `cloudflare-sync` に `events` / `attendance` / `carpool` 系テーブル＋公開／管理エンドポイント |
| 配信 | GitHub Pages（既存） |
| 通知 | LINE 手動＋催促テキスト → 必要なら既存 Web Push をスタッフ向けに転用 |

## 5. Phase 1 要件メモ（出欠）

### データ（案）

実装時はマイグレーション SQL を用意（実行は運用者が行う想定）。

- `activities` … 活動日、開始、場所、種別（練習／試合）、メモ、cohort  
- `attendance_responses` … activity_id、member_id、status（in/out/maybe）、comment、updated_at  
- `attendance_publish` … share_id、activity_id（または cohort 直近）、status  

### 画面（案）

1. **スタッフ：活動一覧／作成**（既存ツール内の新 STEP または別タブ）  
2. **スタッフ：回答状況**（出席・欠席・未定・未回答、催促コピー）  
3. **保護者：公開ページ**（活動情報＋子供の回答ボタン）  

### セキュリティ（最低限）

- 管理 API は現行どおり Bearer（HTML 埋め込みの限界は既知・後回し課題）  
- 公開 API は shareId の推測困難性＋必要なら短時間レート制限  
- 出欠 URL を知っている人が他選手を選べる設計になりやすい → **選手一覧を出す範囲**と、将来の **家庭 PIN** を Phase 1.5 候補に  
- 個人情報は最小（成績・写真は出さない）

## 6. Phase 2 要件メモ（配車）

- 活動に紐づく `carpool_vehicles`（ドライバー、定員、メモ）  
- `carpool_riders`（vehicle、member）  
- `carpool_cargo`（道具 ID or 自由記述）  
- 出欠が `in` のメンバーだけ割当可能  
- LINE 用テキスト：車両ごとの運転・同乗・積み込み  

不公平感の緩和は、まず **「今シーズンの車出し回数」表示**（手動 or 実施確定時カウント）で足りることが多い。最適化ロジックは不要。
