# 小・中学生野球チーム向け活動支援アプリの比較・選定とオリジナルアプリ開発計画

| 項目 | 内容 |
|------|------|
| 文書番号 | TCB-PLAN-001 |
| 版 | 1.0（原案精査・改善版） |
| 作成日 | 2026-07-15 |
| 対象想定 | 小・中学生の野球／ソフトボールチーム運営（スタッフ・保護者の負担軽減） |
| 関連 | 既存ツール [tcb-tools](../README.md)（道具割り振り・LINE展開・PWA） |

本ドキュメントは、市販アプリの比較・導入優先度と、Cursor を活用したオリジナル機能の要件をまとめたものです。  
**原案の方向性（「連絡は LINE」「市販で足りるところは買う／痒いところだけ自作」）は妥当**です。以下は精査結果と改善案です。

## 0. 精査サマリー（評価）

### 総合評価

| 観点 | 評価 | コメント |
|------|------|----------|
| 問題設定（週末負担の削減） | ◎ | 出欠・連絡・スコア・配車・集金・備品の切り分けが実務的 |
| 導入優先度の並び | ○ | 1位（出欠・連絡）が最大効果、という判断は正しい |
| TeamHub／PLAY の機能整理 | ○ | 大枠は公式と一致。一部は過剰簡易化・重複過小評価あり（§2） |
| 「市販にない自作機能」の選定 | △ | **一部は PLAY／既存 tcb-tools と重複**。差別化の再定義が必要（§3） |
| 開発スタック提案 | △ | 緑地開発としては妥当だが、**現行 tcb-tools（静的 HTML＋Cloudflare）と二系統化しやすい** |
| MVP の切り方 | ×（原案） | 機能を一気に列挙しており、初回スコープが広すぎる |
| セキュリティ・未成年データ | ×（原案） | 成績・写真・所在（備品所持）を扱う場合の前提が未記載 |

**結論（推奨スタンス）**

1. **連絡・出欠・チャット・汎用スコアは市販（TeamHub／PLAY 等）＋ LINE を主軸**にする。  
2. **自作は「市販が弱い／チーム固有の運用」に限定**し、まず 1 機能から正式運用まで持っていく。  
3. 自作の第一候補は、既存資産との連続性から **「ビジュアルメンバー表 → LINE 共有テキスト／画像」**、または **「道具割り振り（既存）＋備品ステータスの拡張」**。スコア本丸や球数制限フル実装は後段。

## 1. 市販活動支援アプリの比較と導入優先度（改善版）

保護者・スタッフの「週末の負担」を減らす効果が大きい順です。  
※「ツール例」は代表であり、チームの既存慣習（特に LINE）を尊重すること。

| 順位 | カテゴリ | 代表ツール例 | 主な効率化効果 | 市販／自作の目安 |
|------|----------|--------------|----------------|------------------|
| 1 | スケジュール・出欠・一斉連絡 | TeamHub／PLAY, BAND, LINE・LINE WORKS | 出欠回収・未回答催促・急な連絡の到達管理 | **原則市販**（LINE 併用可） |
| 2 | デジタルスコア・成績 | PLAY by TeamHub, スコアラー 等 | 紙スコア撤廃、自動集計 | **原則市販**（自作は特別ルール不足時のみ） |
| 3 | 配車（車出し）調整 | 調整さん, スプレッドシート,（TeamHub の同行・車出し集計） | 不公平感の可視化・積み込みの共有 | 市販 or 表運用。専用自作は低優先 |
| 4 | 集金・会計 | PayPay 等, スプレッドシート | 現金リスク低減・開示 | **市販／表**（自作非推奨） |
| 5 | 備品・引き継ぎ | TeamHub 用具管理, ドライブ, **既存 tcb-tools** | 所在・担当・次年度引き継ぎ | **既存道具割り振りを優先活用**。不足分だけ拡張 |

### 各カテゴリの詳細理由（原案を補強）

- **スケジュール・出欠**：毎週発生する「未回答への個別催促」が、時間・精神コストの最大源。ここを市販で潰すのが投資対効果が最も高い。  
- **スコア・成績**：紙スコアの忌避と日曜夜集計を減らす。ただし PLAY は少年野球向け機能が既に厚い（§2.2）。  
- **配車**：不満が出やすい役割の可視化。TeamHub 側の「同行人数・車出し可否」と、調整さん／表の併用で足りることが多い。  
- **集金**：キャッシュレスと台帳の可視化が目的。自作会計は監査・トラブル面でコストが大きい。  
- **備品**：東海中央ボーイズ 15 期は既に **道具割り振りツール（tcb-tools）** を本番運用中（LINE 展開・実施確定・交代報告・PWA 等）。市販の「用具管理」と役割が重なるため、**新規アプリでゼロから作らない**。

### 導入判断のチェックリスト（追加）

市販を入れる前に、次を確認する。

1. 保護者の主連絡手段は何か（LINE 一本化か、アプリ追加ログインを許容するか）。  
2. 選手・保護者の個人情報／写真を市販クラウドに載せてよいか（チーム規約・保護者合意）。  
3. 無料プラン制限（広告・ランキング閲覧等）で運用が回るか。足りなければプラスプランの費用対効果。  
4. 「出欠だけ市販・スコアは紙・備品は tcb-tools」のような **ハイブリッド**でよいか。

## 2. TeamHub／PLAY by TeamHub（改善・注記付き）

野球特化の **PLAY by TeamHub** と、共通基盤の TeamHub 機能をまとめたものです。  
料金・制限は公式サイト／ストア表記を基にしています（変動し得るため、導入直前に再確認）。

### 2.1 主要機能

| 領域 | 内容 |
|------|------|
| スケジュール・出欠 | イベント作成、ワンタップ回答、リマインド、同行人数・車出し可否の集計 |
| コミュニケーション | 全体・グループチャット、掲示板（PDF 添付等）、アンケート・日程調整 |
| スコア入力 | Play-by-Play（1 球）、Box Score（イニング簡易）、速報／共有 |
| 成績 | 打撃・投手・守備スタッツの自動集計・ランキング |
| 写真・動画 | アルバム共有（容量・期限の公式表現は要再確認） |
| 用具・対戦 | 共有備品の管理、練習試合募集など |

### 2.2 原案に対する重要な補正

1. **「全員打ち」「リエントリー」等は PLAY 側に既にある程度ある**  
   公式でも草野球向けに「9 人以上の全員打ち」や、ソフトの DP／FP・リエントリー対応が案内されている。  
   → 「ジュニア特有ルール＝全面自作」ではなく、**不足ルールだけ設定／拡張する**前提に変更する。

2. **プラス／エリートの位置づけ**  
   - フリー：基本機能利用可（広告あり等の制限）。  
   - プラス：月額 **2,000 円／チーム（税込）** 目安。スタッツ／ランキング等の制限解除、キャリアスタッツ、ラベル等（表記は時期で差あり）。  
   - エリート：月額 **6,000 円／チーム（税込）** 目安（野球・ソフト向けオプションとしてストア等で案内）。イベントの多条件絞り込み等。  
   → 原案の「フリー＝個人成績上位 5 名まで」は **概ね以前の制限イメージと一致しうるが、現行 FAQ 表記と完全一致しない場合がある**。導入時はアプリ内の制限画面で確認すること。

3. **用具管理は市販にもある**  
   原案の「QR 備品管理」は差別化になりうるが、**単純な備品リスト管理だけなら市販で足りる**可能性が高い。差別化は「車出し・割振り・現地受け渡し」との連動にある。

### 2.3 料金プラン（導入検討用の要約）

| プラン | 目安料金 | 向き |
|--------|----------|------|
| フリー | 0 円 | まず出欠・スコアを試す |
| プラス | 約 2,000 円／月・チーム | ランキング・広告・絞り込み不足を感じたら |
| エリート | 約 6,000 円／月・チーム | 大規模運用・高度なイベント絞り込みが必要なら |

**推奨導入経路**：フリーで 2〜4 週末運用 → 足りない制限だけ有料を検討。最初からエリートは不要なチームが多い。

## 3. オリジナル開発：市販にない／既存と連続する機能（再定義）

前提（原案を維持）：

- **保護者との日常連絡は LINE に任せる**（アプリ内チャットを主戦場にしない）。  
- **天気 API は作らない**（市販アプリや端末天気で足りる）。

### 3.1 推奨機能の再評価

| 原案の機能 | 評価 | 改善後の扱い |
|------------|------|----------------|
| LINE 用テキスト＆画像ジェネレーター | ◎ | **最優先候補**。スタメン・結果・車出し・割振りを LINE 向けに整形。既存 tcb-tools の LINE 展開思想と一致 |
| ジュニア特別ルールのスコア切替 | △ | PLAY のルール対応を先に検証。**不足分だけ**自作 or 運用回避 |
| 球数制限・義務登板間隔アラート | ○（価値高）／難易度高 | 連盟・年度でルール差大。**MVP では手動入力＋簡易アラート**から |
| DnD ビジュアルメンバー表（作戦盤） | ◎ | LINE スクショ／テキスト連携とセットで強い。プロトタイプ向き |
| QR 備品貸出・所持ステータス | ○ | 既存道具割り振りと統合するなら有効。**単独の新アプリは非推奨** |

### 3.2 既存 tcb-tools との関係（追加必須）

現リポジトリには既に次がある。

- 道具の公平割振り・前回ベース最小入れ替え・現地実保有補正  
- LINE／PDF／保護者確認 URL の展開、交代報告、PWA、Cloudflare 同期  

したがってオリジナル計画は次のいずれかにする。

| 方針 | 内容 | 向く条件 |
|------|------|----------|
| **A. 拡張** | tcb-tools に「作戦盤」「汎用 LINE 整形」を足す | 15 期運用を壊したくない・同じメンテ体制 |
| **B. 別アプリ** | Next.js 等で新規。tcb-tools とは API／データ連携 | スコア本丸や他チーム横展開が主目的 |
| **C. ハイブリッド** | 市販＝出欠／スコア、tcb-tools＝備品、薄型 Web＝作戦盤のみ | **現実的な当面案** |

原案のスタック（Next.js＋Supabase＋Vercel）は方針 B 向き。  
**方針 A／C なら、現行の静的 HTML＋Cloudflare Workers／D1＋GitHub Pages の延長がメンテコストが低い。**

### 3.3 MVP フェーズ（原案からの最大改善点）

一気に 5 機能を作らず、次の段階にする。

#### Phase 0（決定のみ・実装なし）

- 出欠／連絡を LINE のみ継続するか、TeamHub／PLAY を試すか決定。  
- 自作は方針 A／B／C のどれかを決定。

#### Phase 1（試作・2〜3 画面）

- **ビジュアルメンバー表（DnD）**  
- **LINE 用テキスト生成**（絵文字付き定型）  
- 配置の画像書き出し（html2canvas 等）または「スクショして送る」導線  
- ローカル保存のみでも可（まず現場で使えること優先）

#### Phase 2（現場ルール）

- チーム固有の守備位置ラベル、人数（9〜12 等）、全員打ち用の打順拡張表示  
- メンバーマスタ（氏名・学年・背番号）— 可能なら tcb-tools マスタと揃える

#### Phase 3（投手管理・簡易）

- 登板日・球数の**手動記録**  
- 休養日数・上限の**簡易アラート**（ルール表は設定 JSON で差し替え可能に）

#### Phase 4（備品ステータス）

- 既存割振り結果に「いま誰が持っているか」を紐付け  
- QR は「道具マスタの URL／ID」を印刷物に載せる程度から（カメラアプリ＋ブラウザで足りることが多い）

#### 後回し（当面やらない）

- アプリ内チャット、会計本丸、配車最適化 AI、天気、Play-by-Play フルスコアエンジンの自作

### 3.4 開発開始プロンプト例（改善版）

方針 C（薄い作戦盤 PWA）の例：

```text
少年野球チーム向けの薄い PWA を作りたい。連絡・出欠は LINE / 市販アプリに任せ、
本アプリは「スタメン・守備位置のビジュアル編成」と「LINE 貼り付け用テキスト生成」だけに絞る。

技術: （候補A）現行 tcb-tools と同様の静的 HTML + 素の JS + PWA
     （候補B）Next.js + Tailwind + PWA
データ: まずは localStorage。メンバーは JSON で差し替え可能に。
画面:
1) メンバー一覧（背番号・氏名）
2) グラウンド図上の DnD 守備配置＋打順
3) 「LINE用テキストをコピー」「配置を画像保存」

まずは画面構成案と、DnD 配置画面のプロトタイプをください。
スマホ片手操作を最優先し、カードだらけのダッシュボードにはしないこと。
```

## 4. リスク・制約（原案への追加）

| リスク | 内容 | 緩和 |
|--------|------|------|
| スコープ肥大 | スコア＋球数＋QR＋作戦盤を同時着手 | Phase 1 のみ正式リリース対象にする |
| 二系統化 | Next.js 新アプリと tcb-tools の二重メンテ | 方針 A／C を優先検討 |
| 未成年データ | 氏名・成績・写真・所在 | 最小データ、公開範囲、保護者合意、トークンの Git 載せ禁止（既存方針踏襲） |
| ルール変動 | JSBB 等の球数・休養は年度改定あり | ルールをハードコードせず設定ファイル化 |
| モバイル DnD | タッチでのドラッグは実装難度が高い | まず「タップで位置選択」代替も用意 |
| 市販との競合説明 | 「TeamHub があるのに別アプリ？」 | LINE 整形・作戦盤など用途を一文で説明できること |

セキュリティは既存リポジトリ方針と同様、**運用開始時点で全部を完璧にしない前提**でも、次は最低限意識する。

- API トークンをリポジトリ平文に載せない  
- クライアントに埋め込んだ秘密は漏洩前提  
- 成績・個人の URL は推測困難な ID／認証を検討
